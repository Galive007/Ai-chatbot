# 06 - Conversation Engine

> **Document Version:** 1.0
>
> **Purpose**
>
> The Conversation Engine is the central coordinator of the entire application.
>
> It receives every incoming message, analyzes the current conversation, coordinates all other systems, and manages the complete conversation lifecycle.
>
> **Important:**
>
> The Conversation Engine **does not generate AI responses**.
>
> It decides **how the conversation should progress**, while other systems determine **who replies**, **what they remember**, and **how they respond**.

---

# 1. Philosophy

The Conversation Engine acts as the **brain of the application**.

It never makes AI decisions itself.

Instead, it coordinates specialized subsystems.

Think of it as the project manager.

Every incoming message must pass through the Conversation Engine before any AI participant is allowed to respond.

---

# 2. Responsibilities

The Conversation Engine is responsible for:

* Receiving new messages
* Validating conversation state
* Updating active topics
* Detecting mentions
* Building conversation context
* Calling the Behavior Engine
* Calling the Memory Engine
* Calling the Prompt Builder
* Scheduling AI responses
* Managing response order
* Synchronizing storage
* Updating conversation summaries

It is **not responsible** for:

* Personality
* Mood
* Prompt writing
* API communication
* UI rendering

# 6. Context Builder

Before any AI is called, the engine constructs a conversation context.

The context contains:

* Current topic
* Recent messages
* Mention information
* Conversation summary
* Current participants
* Active speaker
* Conversation statistics

This context becomes the input for later systems.

---

# 7. Topic Detection

Every new message updates the active topic.

Example

```
Calculus

вҶ“

Derivatives

вҶ“

Integration

вҶ“

Exam Preparation
```

The engine should smoothly transition between topics instead of abruptly replacing them.

The Conversation Engine only tracks topics.

Understanding them belongs to the AI.

---

# 8. Mention Detection

Supported format:

```
@Alex

@Mia
```

The engine extracts mentioned participants.

Example

```
@Alex explain SHM
```

Output

```
Mentioned:

Alex
```

Mention detection must never rely on AI.

It should be deterministic.

---

# 9. Conversation Context

Every AI request receives structured context.

Example

```
Conversation Summary

+

Current Topic

+

Recent Messages

+

Mentioned Users

+

Active Participants
```

The Conversation Engine prepares this information.

# 10. Conversation State

The engine continuously maintains runtime state.

Example

```
Current Topic

Current Speakers

Typing Participants

Recent Activity

Conversation Summary

Last Message

Current Queue
```

This state exists independently from storage.

---

# 11. Response Queue

The engine never immediately generates AI responses.

Instead it creates a response queue.

Example

```
Alex

вҶ“

Mia

вҶ“

Noah
```

The queue is generated by the Behavior Engine.

The Conversation Engine only executes it.

---

# 12. Conversation Lock

Only one conversation pipeline may execute at a time.

Example

```
User sends message.

вҶ“

Conversation Lock

вҶ“

Process

вҶ“

Unlock
```

This prevents race conditions.

---

The Conversation Engine never sends all API requests simultaneously.

Instead:

```

вҶ“

Selected Participants

вҶ“

Schedule

вҶ“

вҶ“

вҶ“

```

This creates realistic timing.

---

# 14. Conversation Queue

If another user message arrives while responses are still being generated:

```
Message A

вҶ“

Processing

вҶ“

Message B

вҶ“

Queue
```

The engine processes them sequentially.

No conversation should be lost.

---

# 15. Interruptions

Real conversations are dynamic.

Example

Alex is typing.

Before Alex sends,

User sends another message.

The engine should determine whether:

* Continue existing reply.
* Cancel reply.
* Regenerate reply.

This prevents outdated responses.

---

# 16. AI-To-AI Conversations

Participants may respond to each other.

Example

```
Alex

вҶ“

Noah

вҶ“

Mia
```

These conversations still pass through the Conversation Engine.

The engine ensures order.

---

# 17. Duplicate Prevention

The engine should avoid duplicate responses.

Example

Alex

"I agree."

Mia

"I agree."

The Prompt Builder should receive enough context to encourage varied responses.

---

# 18. Conversation Summary

Large conversations should not repeatedly send the entire history.

Instead:

```
Conversation Summary

+

Recent Messages
```

The Conversation Engine updates summaries whenever appropriate.

Summary generation belongs to the Memory Engine.

---

# 19. Error Recovery

If an AI request fails:

Conversation continues.

Possible actions:

* Retry
* Skip participant
* Continue with remaining responses

One failure should never stop the conversation.

---

# 20. Cooldown Integration

The engine asks the Behavior Engine:

```
Who is allowed to reply?
```

It never calculates cooldowns itself.

---

# 21. Relationship Integration

The engine passes participant relationships into the prompt.

It does not calculate them.

---

# 22. Typing Integration

The engine schedules typing events.

Example

```
Alex

Typing...

вҶ“

Reply

вҶ“

Stop Typing
```

Typing duration should match message length.

---

# 23. Storage Integration

After every completed conversation:

Persist:

* Messages
* Summaries
* Memories
* Agent states

Storage implementation remains isolated.

---

# 24. Provider Independence

The Conversation Engine never imports Gemini or Groq SDKs.

Instead

```

вҶ“

Provider Manager

вҶ“

or

Groq
```

Changing providers should require zero modifications here.

---

# 25. Scalability

Future versions may support:

* Multiple conversations
* Multiple users
* Real-time rooms
* Multiplayer

The Conversation Engine should remain reusable.

---

# 26. Responsibilities

The Conversation Engine coordinates:

вң“ Chat System

вң“ Behavior Engine

вң“ Memory Engine

вң“ Prompt Builder

вң“ AI Provider

вң“ Storage

It owns orchestration.

---

# 27. Forbidden Responsibilities

The Conversation Engine must never:

* Decide personality
* Decide mood
* Generate prompts
* Store files directly
* Render UI
* Access IndexedDB directly
* Call provider SDKs directly

Keeping responsibilities isolated improves maintainability.

---

# 28. Future Compatibility

Future features should integrate without replacing the engine.

Examples:

* Voice conversations
* Image understanding
* Group rooms
* Authentication
* Multiplayer
* Live collaboration

The orchestration model should remain unchanged.

---

# 29. Success Criteria

The Conversation Engine is considered successful when:

* Every message follows the defined lifecycle.
* No race conditions occur.
* Conversations remain coherent.
* AI responses are scheduled naturally.
* Context is consistently maintained.
* Storage remains synchronized.
* Other engines remain independent.
* AI providers can be replaced without modifying orchestration logic.

The Conversation Engine should behave like an invisible conductor, coordinating every subsystem while remaining independent of the implementation details of those systems.
